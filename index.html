AngularJS

  • AngularJS is a client-side web application framework written in Javascript for use in most web browsers
  • It provides an implementation of model-view-controller in the Web browser to make it easier to produce modular web clients
  • These client-side apps are designed to live on top of JSON-based RESTful web services.

AngularJS is not
for the faint of heart

  • Once we move beyond simple examples, it gets complicated (fast)
  • However, it is VERY powerful
    • You just have to learn its core concepts and idioms.

Core Concepts (1)

  • Data Bindings
    • The value of an HTML tag can be associated with a model object. When one changes, Angular updates the other automatically.

Core Concepts (2)

  • Controllers
    • Controllers are associated with a portion of your HTML and define all of the state and methods that can be accessed within that section of the page.
    • With controllers, you can modularize your web app and decompose data and functionality into small, manageable chunks.

Core Concepts (3)

  • Services
    • Controllers are used to manage data for some portion of a page while it is being displayed. As you move from page to page, controllers will come into and go out of existence all the time.
    • If you need to maintain state between invocations of a controller or if you need to share state between two different controllers, you can create a service.
    • Services are created when an Angular app is initialized
      • They will remain in place for the entire life of the application.

Core Concepts (4)

  • Directives
    • Directives are ubiquitous in Angular; their primary use is to allow Angular to integrate into HTML in a natural way
    • But, they can also be used to create reusable components that combine controllers, data, and HTML
      • You can create a login form component, for instance, that can be re-used across multiple projects.

Core Concepts (5)

  • Embeddable
    • Angular can control as much or as little of a web page as you specify
    • It is easy to embed a small Angular component into an existing page and then incrementally add new functionality over time

Core Concepts (6)

  • Injectable
    • Angular makes use of a concept called dependency injection
    • Rather than use a main routine to wire everything together, Angular controllers and services declare their dependencies up front.
    • It is then up to the Angular run-time system to locate those dependencies at run-time and inject them into the component that needs them.

Modules (1)

  • A module is the primary way to package up a set of controllers into an Angular application
  • To create a module, give it a name and list its dependencies

angular.module('contactsApp', [])
  • This creates a module called contactsApp; this module has no dependencies.

Modules (2)

  • Once you have created a module, you can gain a handle to it by calling angular.module with no dependencies

angular.module('contactsApp')
  • This simply returns the previously created module, so you can add to it.

Modules (3)

  • Once you have a module defined in Javascript, you can tell Angular where it lives in the html with the ng-app directive.
    • In new files, that can be the html tag itself
    • In existing files, you might choose to narrow its scope and put the directive on a div tag

<html ng-app="contactsApp">
</html>

Controllers (1)

  • To actually do something in Angular, you need a controller
  • Controllers are declared using the controller function of an Angular module.

angular.module('contactsApp')
  .controller('MainController', [<dependencies and code>])
  • The second parameter is an array that allows the controller to declare its dependencies.

Controllers (2)

  • A simple controller may have no dependencies

<MODULE>.controller('MainController', [function() {
  var self    = this;
  self.name   = "Ken Anderson";
  self.update = function() {
    self.name = "Kenneth M. Anderson";
  };
}]);
  • Here we create a controller that manages one piece of state (self.name) and provides access to one method
  • Anything defined on this is available to the HTML that makes use of the controller.

Controllers (3)

  • More complex controllers will declare dependencies

<MODULE>.controller('MainController', ['$http', function($http) {
  var self    = this;
  self.name   = "Ken Anderson";
  self.update = function() {
    return $http.get('/api/1.0/update_name').then(function(response) {
      self.name = response.data.new_name;
      return response;
    });
  };
}]);
  • Here we create a controller that requires the use of Angular's built-in HTTP module.
  • The array is expected to have a set of strings and one function at the end. The function takes one parameter per dependency; each parameter will have the same name as the dependency.

AngularJS: Hello World


<!DOCTYPE html>
<html>
  <head>
    <title>Hello World</title>
  </head>
  <body ng-app>
    <h1>Hello {{name}}</h1>
    <input type="text" ng-model="name" placeholder="First Name">
    <script src="http://ajax.googleapis.com/ajax/libs/angularjs/1.3.11/angular.min.js"></script>
  </body>
</html>

Basic HTML5 document structure

Directives: ng-app on the body tag; ng-model on input tag

Script tag at the bottom loads Angular via Google's CDN

What does it do?

  • The input tag declares that a model object exists called name
  • The heading used Angular's built-in syntax to reference the values of models
    • {{name}} pulls in the current value of name and inserts it into the HTML as if it had been placed there originally
  • This is an example of a one-way data binding
  • Any changes to the input field get communicated to all references automatically.

First Controller


<!DOCTYPE html>
<html ng-app="hello">
  <head>
    <title>Hello Controller</title>
  </head>
  <body ng-controller="HelloController as c">
    <p>The message is: <input size="30" type="text" ng-model="c.message"></p>
    <p>The message is: {{c.message}}</p>
    <p><button ng-click="c.changeMessage()">Change Message</button></p>
    <script src="http://ajax.googleapis.com/ajax/libs/angularjs/1.3.11/angular.min.js"></script>
    <script>
      angular.module('hello', []).controller('HelloController', [
        function() {
          var self = this;
          self.message = "Hello from the Controller";
          self.changeMessage = function() {
            self.message = "Goodbye from the Controller";
          }
        }
      ]);
    </script>
  </body>
</html>

Explanation (1)

  • First, declare that Angular controls the entire document

<html ng-app="hello">
  • Second, load Angular via the CDN

Explanation (2)

  • Third, indicate what portion of the document is controlled by our controller

  <body ng-controller="HelloController as c">
  • This syntax says
    refer to an instance of the HelloController via the variable c

Explanation (3)

  • Fourth, bind elements of the HTML to the controller

<p>The message is: <input size="30" type="text" ng-model="c.message"></p>
<p>The message is: {{c.message}}</p>
<p><button ng-click="c.changeMessage()">Change Message</button></p>
  • The input field is bound to the message variable.
  • The paragraph includes the value of the same variable
  • The button binds its click event to a method.
  • The binding is two-way. Changes propagate.

Explanation (4)

  • Fifth, create the controller

angular.module('hello', []).controller('HelloController', [
  function() {
    var self = this;
    self.message = "Hello from the Controller";
    self.changeMessage = function() {
      self.message = "Goodbye from the Controller";
    }
  }
]);
  • Here we create the controller in-line; we could also do this in a separate file and include that file with a script tag

Explanation (5)

Why did we do this?


var self = this;
  • Closures!
  • Note that we reference self.message in the function changeMessage().
  • If we had said this.message in changeMessage(), there is a chance that this no longer refers to the controller.
  • So we store the pointer to the controller in self and a closure captures that value when the changeMessage() function is created.
